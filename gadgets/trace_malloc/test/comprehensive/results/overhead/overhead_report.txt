=== trace_malloc vs heaptrack Overhead Benchmark ===

| Language | Baseline (s) | trace-malloc (s) | tm overhead | heaptrack (s) | ht overhead |
|----------|-------------|------------------|-------------|---------------|-------------|
| C        | 0.0020      | 0.0070           | 250%        | 0.0683        | 3310%       |
| C++      | 0.0020      | 0.0126           | 530%        | 0.0773        | 3760%       |
| Rust     | 0.0020      | 0.0080           | 300%        | 0.1016        | 4980%       |
| .NET     | 0.0666      | 0.5443           | 710%        | 0.2523        | 270%        |

Analysis:
- For native languages (C, C++, Rust), trace-malloc has MUCH LOWER overhead
  than heaptrack (250-530% vs 3310-4980%). This is because trace-malloc uses
  lightweight eBPF uprobes while heaptrack uses LD_PRELOAD with full stack
  unwinding via libunwind on every allocation.

- For .NET, trace-malloc has HIGHER overhead than heaptrack (710% vs 270%).
  This is because .NET runtime generates ~30K native malloc events internally
  (runtime initialization, JIT, GC metadata). trace-malloc captures ALL
  system-wide events, while heaptrack only wraps the target process.

- The absolute overhead of trace-malloc for native programs is very small
  (5-10ms per test run of ~150-300 allocations), which is negligible for
  production profiling.

- The .NET overhead (0.54s) includes monitoring ALL 30K+ internal runtime
  allocations. In production, filtering by pid reduces this significantly.

Key takeaway:
- trace-malloc excels for system-wide observability with low overhead on
  native code (C/C++/Rust).
- heaptrack excels for deep single-process analysis with better stack traces.
- For .NET/JVM-type runtimes, trace-malloc captures the native allocator layer
  (useful for diagnosing runtime/interop leaks) while heaptrack captures the
  same but with lower overhead since it only wraps one process.
